How this blog was made

developer • Aug 01, 2020

Building this blog went through multiple iterations.  The objective being sought after was ease-of-use combined with the ability to be hosted as a static site.  

Network Diagram

Choosing the Medium

When looking into possible solutions I knew right away that ease-of-use had to be involved as the primary objective here is to allow my wife to make posts.  With that said, I did look into multiple platforms.

WordPress: I used to do work/blog on a WordPress platform, and while I found the ease-of-use to be there, I never quite liked the UX (and plugin system overload).  So I ruled this one out.

Gatsby + GitHub pages: As I started down the path of scaffolding a blogging application out, and started writing the first post, I saw the folder-structure and markdown articles and said "ok, maybe I have to teach the wife a few things".  Then I got to the deployment and thought "Junior engineers struggle with Git, teaching my wife to regurgitate commands shouldn't be an option.  Maybe a script, or do I scrap the idea?"

Ghost: I first heard about Ghost about a year ago, it being open-source is a plus.  But hosting costs money, and after digging around on the net I found a GitHub project for ghost-on-GitHub as a static site, which checks my speed of hosting box!  So I started down this path...

Installation

Installation on my home infrastructure was a cinch as most pieces are dockerized. So in order to make it work, I did as all good engineers do, and followed in the footsteps of someone else (see ghost-configuration-guide from the resources). I feel as though I'd be doing a disservice by attempting to regurgitate an already good guide for the installation steps of the pieces.  The only part not captured by the guide is setting up my ghost-to-github-pages docker image.  

Ghost to GitHub pages:

How it works.

This section will more-or-less be going over the following network diagram.

both the diagram and our blogging adventures start with the family navigating to the ghost application in the browser via an internal URL.  behind the scenes the request is sent to our local server and filtered through the Nginx config.  the Nginx config, which works as a reverse proxy, intercepts the subdomain and forwards the request to ghost running on that server.  Once the request has made it to Ghost, then Ghost can simply behave as normal and serve up its user experience and allow us to write our blog pages.

Noticed from the diagram viewer traffic never actually touches our infrastructure and instead all get filtered through Github.  That is because this blog is 100% hosted as static pages from Github (check out the repo from the resources below! make this a hyperlink...).  The dockerized cronjob is the magic glue to make this happen.

The dockerized job is a simple script. The script takes the following actions:

  1. Sets up a local git user profile
  2. Clones the GitHub repo where the static site will be hosted using a Personal Access Token
  3. Removes all non-git-related files, and adds the CNAME file for GitHub  (this is to have clean commits).
  4. Use the ghost-static-site-generator npm module to generate the static site.
  5. Push up the new changeset

The containerization and scheduling of those 5 steps allow the automation of a static-site generator and easy-to-use blog.
